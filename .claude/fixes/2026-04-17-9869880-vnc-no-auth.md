# VNC Spinner — noVNC Never Connects

**Date:** 2026-04-17
**SHA:** 9869880
**Symptom:** Navigating to `https://mcomz.local/vnc/vnc.html?path=vnc/websockify&autoconnect=true` showed a loading spinner indefinitely. No password dialog appeared.

## Diagnosis

SSH + journalctl revealed:

```
websockify: Plain non-SSL (ws://) WebSocket connection
websockify: connecting to: 127.0.0.1:5901
Xvnc: Connections: Accepted: 127.0.0.1::38018
Xvnc: VNCSConnST:  Closing 127.0.0.1::38018: Clean disconnection
Xvnc: EncodeManager: Framebuffer updates: 0
```

The WebSocket and TCP connection to Xvnc were both completing successfully. "Clean disconnection" + 0 framebuffer updates means authentication never finished.

Root cause: Xvnc was configured with `-SecurityTypes VncAuth -rfbauth ~/.vnc/passwd`. noVNC receives the VncAuth challenge, fires a `credentialsrequired` event, and renders a password dialog — but the dialog appears behind/under the loading spinner overlay, and is invisible to the user. The user sees only the spinner, which never clears because the password is never submitted. After a timeout or browser tab change, noVNC closes the connection cleanly.

Confirmed by direct TCP test: `\x01\x02` = 1 security type, type 2 (VncAuth).

## Fix

Switched Xvnc to `-SecurityTypes None` and removed `-rfbauth`. Port 5901 is not network-accessible — it is only reachable via `nginx → websockify → 127.0.0.1:5901`. The nginx reverse proxy is the security boundary. VncAuth adds friction without providing security.

**site.yml change:**
```
-/usr/bin/Xvnc :1 -geometry 1280x800 -depth 24 -rfbauth "$HOME/.vnc/passwd" -rfbport 5901 -SecurityTypes VncAuth
+/usr/bin/Xvnc :1 -geometry 1280x800 -depth 24 -rfbport 5901 -SecurityTypes None
```

Applied live: `sed -i` on `/usr/local/bin/mcomz-vnc-start` + `systemctl restart mcomz-vnc`.

Post-fix TCP test: security type `\x01\x01` (None), SecurityResult `\x00\x00\x00\x00` (OK). No password required.

## Outcome

**Confirmed working live (2026-04-17):** noVNC auto-connects without password prompt. JS8Call is already running inside the VNC session (confirmed by `systemctl status mcomz-vnc` showing `js8call` and `/usr/bin/js8` in the cgroup). User needs to open the noVNC URL and confirm JS8Call window is visible.

Smoke test: 74/74 ✅ (WikiMed also confirmed downloaded — 2.1 GB, Restart=on-failure fired once due to DNS delay, completed at 01:17:05 BST on first boot).
